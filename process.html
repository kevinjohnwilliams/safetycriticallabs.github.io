Safety Critical Labs / Certification Process

From assessment
to formal
determination.

A four-phase process that produces a formal determination — not a recommendation, not a risk score — that your AI system was assessed against a defined standard. The language matters. The rigor behind it matters more.

Four phases
01 Applicability determination 02 Evidence package review 03 Technical assessment 04 Certificate issuance
01
Applicability determination
Scope · Classification · Tailoring

Before evidence is reviewed or any technical assessment begins, we establish exactly what is being assessed and at which level. Applicability determination is the phase that defines scope — and it cannot be skipped or combined with evidence review without undermining the integrity of the determination.

System boundary definition — What components of the AI system are within scope? What interfaces with out-of-scope systems require explicit boundary documentation?
Classification assignment — Is this system Safety-Critical, Mission-Critical, or Mission Support? Classification drives which requirement sets (AI-1 through AI-9) are applicable and at what depth.
Tailoring rationale — Where do operational constraints require tailoring of standard requirements? Every tailoring decision is documented with an explicit rationale that becomes part of the certification record.
Applicability determination document — Formal output of Phase 1. Establishes the assessment scope in writing before any evidence is submitted. Client signs off before Phase 2 begins.
02
Evidence package review
Documentation · Test reports · Verification matrices

The client submits their evidence package against the applicable requirement set. Our team reviews each submission against specific framework requirements — with documented findings, not general observations. Every finding is traceable to a requirement number, a specific criterion, and a pass/fail disposition.

Documentation review — Architecture documentation, model cards, training data documentation, validation reports, and operational procedures. Reviewed against each applicable AI-1 through AI-9 requirement.
Test report analysis — ML test coverage results, bias evaluation reports, OOD detection test results, adversarial robustness test outcomes. Assessed against framework criteria, not general best practices.
Verification matrix review — Requirements traceability from system requirements to test evidence. Every AI requirement must have a corresponding verification method and documented test result.
Evidence package review report — Formal output. Documents open items, requests for additional evidence, and preliminary findings. Client has opportunity to address open items before Phase 3.
03
Technical assessment
Structured audit · Findings · Remediation paths

The structured audit of your AI system against the framework requirements. Where evidence review examined documentation, technical assessment examines the system itself — its architecture, its test infrastructure, and the operational validity of its verification claims.

System walkthrough — Technical review of architecture decisions against framework requirements. Focus on AI-specific failure mode coverage, not general software quality.
Test infrastructure review — Assessment of the test environment used to generate evidence. Test infrastructure that cannot reproduce results doesn't support a determination.
Finding grading — Each finding is graded Pass, Conditional Pass (with defined remediation), or Fail. Grading rationale is documented against specific requirement criteria, not assessor judgment.
Remediation paths — For Conditional Pass findings, a defined remediation path with specific acceptance criteria is provided. Remediation is verified before certificate issuance — it is not taken on faith.
04
Certificate issuance
Formal determination · Classification level · Surveillance schedule

A formal determination document stating what was assessed, what was found, and what is required to maintain certificate validity. Every assessment produces a Determination Document regardless of outcome. A certificate is issued only when applicable requirements are met.

Certification Determination Document — Issued regardless of outcome. States the system assessed, framework version, classification level, requirement sets, findings summary, and the formal determination: Certified, Conditionally Certified, or Not Certified.
Certificate (if certified) — States system name and version, client organization, SCL framework version, classification level, requirement sets assessed, issue date, validity period, and surveillance due date. The certificate does not state the system is safe. It states the system was assessed against a defined standard and met the applicable requirements at the time of assessment.
Assessment record retained — All documentation from all four phases is retained by SCL for a minimum of 10 years or the life of the certified system. Available for regulatory review.
Surveillance schedule established — Certification is not a one-time event. AI systems change behavior over time. Every certificate includes a structured surveillance schedule.
Safety-Critical Annual surveillance audit. Full re-assessment of drift metrics, OOD detection logs, and operational performance data since certification.
Mission-Critical Biennial surveillance audit. Focused re-assessment of continuous validation records and any model updates made since the original certification.
Triggered re-assessment Required for any of: model version change, significant operational environment change, defined drift threshold breach, or incident involving an AI-assisted decision with safety consequences.
Assessment timeline

Six to ten weeks,
start to certificate.

Timeline assumes a complete evidence package submitted on time with no major clarification rounds. Complex Safety-Critical systems may take longer. Every phase has a defined output before the next begins.

Phase 01
Applicability determination
1–2 weeks
Output: Signed Applicability Determination Record. Establishes scope, classification, and applicable requirement sets. Both parties sign before Phase 2 begins.
Phase 02
Evidence package review
3–4 weeks
Output: Evidence Review Report with preliminary findings for each requirement. Client reviews open items before the technical assessment begins.
Phase 03
Technical assessment
1 week on-site or remote
Output: Technical Assessment Report with final findings. Major, Minor, and Observation findings graded against specific requirement criteria.
Phase 04
Determination and issuance
1 week
Output: Certification Determination Document issued regardless of outcome. Certificate issued if applicable requirements are met.
6–10
Weeks total, start to certificate. The clock starts when the Applicability Determination Record is signed. The assessment fee covers all four phases through certificate issuance.
What you prepare

The evidence package.
What you bring to Phase 2.

Phase 2 is an evidence review, not a discovery exercise. You submit a structured package against each applicable requirement. SCL reviews against specific framework criteria, not general best practices. Every finding is traceable to a requirement number.

An incomplete or placeholder submission resets the Phase 2 clock. The Evidence Package Guide provided at the end of Phase 1 details exactly what is required for your classification level.

Not ready to certify yet? Organizations can request a readiness assessment before committing to a full certification cycle. A readiness assessment identifies specific gaps against the framework without producing a formal determination.
System documentation Architecture documentation, model cards, training data records with version control and provenance documentation.
Test reports ML test coverage results, bias evaluation reports, OOD detection test results, adversarial robustness outcomes. Dated and reproducible.
Validation records Post-deployment performance data, drift monitoring logs, continuous validation records per AI-4 requirements.
Verification matrix Requirements traceability from each AI-1 through AI-9 requirement to a corresponding verification method and documented test result. No placeholders.
Operational procedures Human override procedures, authority limit documentation, OOD escalation protocols, and human-AI interaction logs per AI-9 requirements.
